DatabaseSchema.txt
------------------

This file documents the structure of the MetaView statistics database.
For details concerning the use of the information in this file please consult
the 'MetaView Statistics database integration' chapter of the MetaSphere
Enhanced Applications Server, CommPortal and MetaView Reporting manual. 

==================
DATABASE STRUCTURE
==================

The 'public' schema contains Statistics Database for the local MetaView Server
(the server that you've connected to).

There is also a 'cluster' schema, which allows access to the same data as the
'public' schema, but aggregated across a cluster of MetaView Servers.  This can
be used to make cluster wide queries, e.g. for a clustered CFS deployment.
Each view in the cluster schema has a sitename column added. This column should be
queried to find out which site in the cluster the statistics are for.

The database is comprised of tables for each type of object on which statistics
are stored.  For each object type, there are three tables, whose name is the
object type plus one of the following:

-  _FIVEMINUTES
-  _ONEHOUR
-  _ONEDAY

These tables store statistics data summarized over five minutes, one hour and
one day respectively.  So, the ISUP_MEDIA_CHANNEL_FIVEMINUTES table contains
statistics for all ISUP Media Channels, summarized into five-minute periods.
Note that tables for MetaSphere EAS objects are prefixed with 'SDP' (not 'EAS').

==============
ACCESS DETAILS
==============

Database software: PostgreSQL 13.10
Port: 5432
Name: ems_stats_db
Username: statsread
Obtain the password from your secure administrator, who can configure the password in the 'Configure database passwords' craft option.

Access is read-only.

===============
TABLE STRUCTURE
===============

Each row within these tables contains the statistics for one object of the
appropriate type within one time interval.  For example, each row in the
SUBSCRIBER_GWAY_ONEDAY table contains statistics for a particular Subscriber
Gateway summarized across one particular day.

The columns in the tables are as follows.

-  A STATISTICTIMESTAMP column.  This tells you the time at which the interval
   covered by this row begins. It is of SQL type TIMESTAMP.
-  One or more identifier columns.  The combination of these columns tells you
   which object (e.g. which Subscriber Gateway) the row refers to.
-  One or more statistic columns, containing the value of that statistic in
   this interval.
-  An ESTIMATED column: if 'N', the statistics in this row are guaranteed to be
   accurate; if 'Y', the statistics are only an estimate.

The rest of this file details the columns in each table.
Statistics are categorized into 5 types, event-rate, regular, minimum, maximum
and event-based.
The column details give the type, units and SQL data type of the column.
Column details for event-based statistics also give the column names of the
sample sum and event count columns from which the event-based statistic is
calculated.
Note that the identifier for MetaSphere EAS objects is 'SDP_SERVER_NAME'
(not 'EAS_SERVER_NAME').

Also note that the first identifier for CFS / UMG / IS / MVD objects is
'METASWITCH_NAME' and the table for Call Feature Server stats is the
CALL_AGENT table.

============================
NOTES ON SPECIFIC STATISTICS
============================

The following statistics have quirks that are worth noting.

1. MEDIA_GATEWAY : Traffic Information:
   - Calls currently connected
   - Total call attempts
   - Successful call attempts
   - Failed call attempts
   - Calls using T.38 fax encoding

   Ordinarily, media gateway calls will use a single H.248 context and these
   statistics are as you would expect.
   But if your system uses multiple H.248 contexts per call, then because
   these statistics actually count operations with H.248 contexts, the values
   reported will in fact be a multiple of the number of actual calls - except
   for failed call attempts, since it's unlikely that multiple contexts will
   fail within a single call attempt.
